The idea of this project was first briefly outlined in these conference proceedings . . . PDMU . . . and later a somewhat expanded version was published on . . . ArXive.org . . .

It is easy to understand that the best protection is the STR-strategy (specific typing rhythm). You can verify this by following this link . . . but to use the STR-strategy, the user must adhere to a pre-memorized secret rhythm when typing the password, so now we are considering a more user-friendly strategy, when he does not need to remember the secret rhythm, but type the password as he usually does in his natural typing rhythm (NTR). The program must learn to recognize this rhythm (something similar to the fist of a radio operator) using relevant statistical data, but it is still necessary to distinguish this rhythm from the rhythm of the attacker, who will type the password in a 'common' rhythm. Now in this version a simple check is used (more for demonstration purposes) in which the time of pressing each key should be within the range from the minimum to the maximum of the corresponding values obtained from the statistical values (which are established during the program's training phase): 


const algo_name = "min-max verification";  // Please provide the name of your algorithm

function checker(stats, arr){
	for(let j=0; j < stats[0].length; j++){
		const column = getColumn(stats, j);
		if((arr[j] <  Math.min(...column)) || (arr[j]  >  Math.max(...column))){
			return false;
		}
	}
	return true;
}

With proper training (i.e., with sufficient statistical data), this method begins to yield good — though not perfect — results. Most importantly, the method is quite robust against cracking:

The most interesting challenge for any researcher in this topic is to propose their own method for recognizing a user's natural typing rhythm. But — and we must stress this again — it is crucial to distinguish the user's rhythm from that of an attacker, who would likely type the password in some generic "common" rhythm. Anyone is welcome to fork this project and modify the function checker(stats, arr) recognition function for NTR. If your function demonstrates statistically good results, the project author will, upon your request, add a link to your proposed fork in the following table:
